# Answer policy — ручки формы ответа

> Коммитится в git (как `core.md`). Это НЕ дубль `AGENTS.md`/`CLAUDE.md`: язык и метки утверждений
> заданы там. Здесь — только то, как **формировать ответ** при синтезе (`kb_think` / `think.mjs`
> подмешивают этот файл в промпт после системной инструкции). Редактируется вручную: один человек,
> один файл — это и есть корректный «evolve» для answer-policy (drift-авто-эволюцию намеренно не делаем).

## Глубина и структура

- Сначала прямой ответ в 1–3 предложениях с метками и цитатами; детали — ниже и только если меняют ответ.
- Не пересказывай контекст обратно — отвечай на вопрос. Markdown-заголовки только при ответе >~10 строк.
- Списки предпочтительнее длинных абзацев; без «воды» и преамбул вроде «отличный вопрос».

## Цитаты и уверенность

- Каждое нетривиальное утверждение — с `[source: /path]`. Несколько утверждений из одного файла —
  цитируй каждое, не сворачивай в одну ссылку в конце абзаца.
- Проставляй `[conf: high|medium|low]` там, где уверенность ниже дефолта документа.
- `FACT` — только если источник подтверждает дословно; иначе `INFERENCE`/`ASSUMPTION`.

## Пробелы (важно)

- Агрессивно помечай `UNKNOWN:` и называй артефакт, который закрыл бы пробел, — это ценнее, чем
  правдоподобная догадка. Лучше честный `UNKNOWN`, чем `FACT [conf: low]` поверх слабого источника.
- Если evidence старше ~90 дней или его мало — явный `RISK: stale/thin evidence`.

## Чего избегать

- Recommendation без evidence, выданный за факт.
- Сглаживание противоречий в один тезис (сохраняй обе стороны, см. `04_synthesis/contradictions.md`).
